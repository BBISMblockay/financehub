App status, architecture, and how SILO works

This repository powers SILO (Silo ERP): mostly static HTML tools for operations, finance, inventory, purchasing, and wholesale workflows. Data lives in Supabase (PostgreSQL). Scheduled jobs and a small Node service move data in from Google Sheets and Shopify. This page is the friendly map for anyone onboarding or debugging the system.

How to use SILO → business workflows (sales verification, AP, POs, AR, inventory) plus sign-in and hubs.

What SILO does

SILO is an internal web suite: dashboards and forms implemented as plain HTML pages with client-side JavaScript. People sign in with Supabase Auth (see pages/login.html and shared pages/config.js for project URL and anon key). After login, pages query tables and views directly with the Supabase JS client, filtered by Row Level Security policies defined in the database.

Separately from the static site, Node.js scripts and a small Express API synchronize operational data: retail-style sales and inventory snapshots from Shopify, and multi-location “Silo” spreadsheets into the same Supabase project so the HTML tools stay consistent.

Big picture architecture

Think of three lanes: human-facing static pages (GitHub Pages or similar), background sync (GitHub Actions + optional hosted API), and Supabase as the single source of truth for app data.

How data flows

Nightly Silo (Sheets → Supabase)

scripts/sync-silo-inventory-sales.mjs reads CSV export URLs from config/silo-sources.mjs (one row per retail location plus online). It upserts inventory and daily sales into tables such as inventory_on_hand and sales_by_day, and can refresh verification summaries. 

flowchart LR
  A[Google Sheets tabs] -->|published CSV URLs| B[sync-silo-inventory-sales.mjs]
  B --> C[inventory_on_hand]
  B --> D[sales_by_day]
  B --> E[sales_verification_store_comp_summary]

Shopify → Supabase

server/index.mjs exposes HTTP endpoints that pull orders, products, variants, locations, and inventory levels from one or more Shopify stores, then writes products_master, sales_by_day, inventory_on_hand (append snapshots), and sync_state per shop. Configure stores via environment variables (SHOPIFY_CONNECTIONS JSON or legacy SHOPIFY_STORE / SHOPIFY_TOKEN). 

flowchart TB
  S[Shopify REST Admin API] -->|orders products variants inventory| N[Express server]
  N --> P[products_master]
  N --> Q[sales_by_day]
  N --> R[inventory_on_hand snapshots]
  N --> T[sync_state]

Accounts receivable (Sheets → Supabase)

server/ar-sync.mjs (run via npm run sync:ar) fetches published CSVs from Google Sheets, normalizes customers and invoices, and upserts into ar_customers, ar_invoices, and job bookkeeping in job_sync_state.

HTML apps and feature areas

Most tools are standalone pages. Root-level files (for example finance.html, inventory.html) and pages/*.html mirror each other in some cases; use whichever path your hosting maps to. Common pattern: Tailwind via CDN, Supabase v2 from CDN, session from Supabase Auth.

Shopify sync service (server/index.mjs)

Package name: silo-shopify-sync. Start with npm start (runs node server/index.mjs). Requires SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY. Optional tuning: SHOPIFY_API_VER, SHOPIFY_PULL_MODE (created vs updated), SHOPIFY_DAYS_BACK, and multi-store SHOPIFY_CONNECTIONS.

RouteMethodPurpose
/GETJSON service banner and route list
/healthGETSimple alive check
/api/sync/shopifyPOSTRun sync; optional JSON body (since, days_back, mode)
/api/sync/shopify/runGETSame sync via query string (handy for cron or manual GET)

The server uses node-fetch with retry on HTTP 429, paginates Shopify “Link” headers, and chunks Supabase writes (500 rows) for reliability.

Scripts and supporting config

  • scripts/sync-silo-inventory-sales.mjs — nightly Silo pipeline; env flags documented in the file header (SILO_SALES_SYNC_MODE, SILO_ONLY_SOURCE, skips, etc.).
  • config/silo-sources.mjs — inventory and sales CSV URLs per location_tag; validated on startup.
  • server/ar-sync.mjs + server/run-ar-sync.mjs — AR import from Sheets; npm run sync:ar.
  • scripts/audit-inventory-product-titles.mjs, scripts/backfill-inventory-titles-from-sheets.mjs — maintenance utilities.
  • scripts/sql/*.sql — ad hoc checks (for example inventory workboard title validation).

Technology summary

  • Runtime: Node.js (ES modules: "type": "module" in package.json).
  • HTTP: Express 4, CORS enabled, JSON body up to 10 MB for sync payloads.
  • Database & auth: Supabase JS v2 (@supabase/supabase-js) for service writes in Node and anon auth on static pages.
  • Integrations: Shopify Admin REST API; Google Sheets as CSV sources (no Sheets API key required when using published export URLs).
  • Parsing: Papa Parse for CSV in AR sync.
  • Frontend: Static HTML; Tailwind and Supabase often loaded from CDNs; no bundler required for deployment.
  • Hosting hint: CNAME points to silo-baseballism.com (GitHub Pages or similar static hosting).

Important notes

  • Two ways into sales/inventory data: Shopify sync service and Silo spreadsheet sync both target overlapping tables; know which job owns “truth” for a given channel (online vs stores).
  • sales_by_day.row_hash: Shopify upserts use a deterministic hash; Silo script uses its own hashing rules — both are designed to avoid duplicate rows for their source.
  • Inventory snapshots: Shopify path inserts new snapshot rows; understand retention or deduplication in your reporting views.
  • Overlap windows: Shopify incremental sync subtracts a small day overlap from last sync time to avoid missing edge orders.
  • Config.js: update pages/config.js carefully; it ships to every user’s browser.
